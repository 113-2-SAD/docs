> 手動測試：數位產品的最後防線

在這個自動化盛行的時代，你可能會想：「為什麼我們還需要手動測試？難道不是所有測試都該自動化嗎？」

其實，這就像是問「為什麼專業廚師還需要親自品嚐食物，而不只依賴溫度計和時間表」一樣。自動化工具固然強大，但有些事情，如美味的體驗、直覺的感受，還是需要人的參與。

讓我們一起來探索手動測試在 SADo 這類軟體開發中的價值。

## 為什麼 SADo 專案需要手動測試？

即使我們為 SADo 建立了完善的自動化測試（Jest 單元測試和 Playwright E2E 測試），以下場景仍然需要手動測試：

### 1. 直覺體驗的評估

想像一下，SADo 的新用戶界面已經開發完成：

- 自動化測試可以告訴你「新增待辦事項」按鈕是否可點擊
- 但只有手動測試能告訴你「這個按鈕是否容易被用戶找到」或「顏色是否舒適」

### 2. 發現未預期的問題

當用戶實際使用 SADo 時，他們可能會：
- 在半途取消新增待辦事項然後立即嘗試排序
- 在填寫長描述時誤點了頁面其他區域
- 嘗試拖曳待辦事項到不同分類

這些「創意」操作常常不在預設的測試腳本中，但手動測試可以模擬這種「用戶亂點」行為，發現潛在問題。

### 3. 敏捷開發的得力助手

在 SADo 的開發初期，界面和功能可能每天都在調整：
- 今天決定待辦項目要顯示截止日期
- 明天又決定要加上優先級標記
- 後天可能要重新設計整個清單視圖

在這種快速變化的環境中，手動測試比維護自動化測試腳本更加高效。

## 手動測試方法：從隨機探索到系統性檢查

### 探索性測試：像偵探一樣思考

探索性測試就像是扮演一個好奇的偵探，不按常規路徑探索 SADo 應用。這種測試沒有固定腳本，完全依靠測試者的創意和經驗。

**實用技巧**：
1. **設定時間盒**：「我要用 30 分鐘探索 SADo 的篩選功能」
2. **角色扮演**：「如果我是一位急著記錄課堂筆記的學生，我會如何使用這個應用？」
3. **邊界思考**：「如果我嘗試新增 100 個待辦事項會發生什麼？」

### 使用者接受度測試：真實用戶的反饋

找幾位同學或朋友（理想情況下是目標用戶群體）嘗試使用 SADo，看他們是否能夠順利完成核心任務。

**SADo UAT 任務範例**：
- 請你新增一個帶截止日期的待辦事項
- 請嘗試將剛才的待辦事項標記為完成
- 請你找出所有已完成的待辦事項
- 請你編輯一個現有待辦事項的描述

觀察並記錄用戶在完成這些任務時的困難點，這些信息對改進 UI/UX 極為寶貴。

## 打造有效的手動測試計畫：SADo 案例

### 測試案例設計

以下是 SADo 專案中一個手動測試案例的例子：

```markdown
# SADo 手動測試案例：待辦事項篩選功能

## 測試 ID
MT-FILTER-001

## 測試目標
驗證用戶能夠使用多種條件組合篩選待辦事項

## 前置條件
- 用戶已登入 SADo 應用
- 系統中已存在多種狀態的待辦事項（已完成、未完成）
- 待辦事項具有不同的優先級和截止日期

## 測試步驟
1. 點擊「篩選」按鈕
2. 選擇狀態「未完成」
3. 選擇優先級「高」
4. 設定截止日期範圍為「一週內」
5. 點擊「應用篩選」

## 預期結果
- 清單中只顯示符合篩選條件的待辦事項：
  * 狀態為「未完成」
  * 優先級為「高」
  * 截止日期在一週內
- 篩選條件顯示在篩選區域
- 待辦事項計數更新為篩選後的數量

## 測試環境
- 瀏覽器：Chrome 最新版
- 螢幕尺寸：桌面版 (1920x1080)
```

### 執行測試並記錄結果

當你執行上述測試案例時，應該使用表格記錄每個步驟的結果：

| 步驟 | 預期結果 | 實際結果 | 通過/失敗 | 備註 |
|------|----------|----------|-----------|------|
| 1. 點擊「篩選」按鈕 | 篩選面板打開 | 篩選面板打開 | ✅ 通過 | 過渡動畫流暢 |
| 2. 選擇狀態「未完成」 | 狀態選項成功選中 | 選中成功 | ✅ 通過 | |
| 3. 選擇優先級「高」 | 優先級選項成功選中 | 選中成功 | ✅ 通過 | |
| 4. 設定截止日期範圍 | 日期選擇器正常工作 | 日期選擇器顯示錯誤 | ❌ 失敗 | 日期選擇器未加載樣式 |
| 5. 點擊「應用篩選」 | 列表更新，符合條件 | 未測試 | ⚠️ 阻塞 | 由於步驟 4 失敗 |

### 提交清晰的缺陷報告

當你發現問題時，需要提交詳細的缺陷報告：

```markdown
## SADo 缺陷報告

### 基本信息
- ID: BUG-FILTER-001
- 標題: 篩選功能中日期選擇器樣式未正確加載
- 優先級: 中
- 嚴重程度: 阻塞功能
- 報告者: 測試團隊成員張小明
- 發現日期: 2023-10-15

### 重現步驟
1. 以測試帳號 test@example.com 登入系統
2. 點擊主頁面上的「篩選」按鈕
3. 嘗試設定截止日期範圍

### 實際結果
日期選擇器出現，但無樣式，顯示為瀏覽器默認的日期輸入框，並且無法選擇日期範圍，只能選擇單一日期。

### 預期結果
應顯示自定義樣式的日期範圍選擇器，允許選擇開始和結束日期。

### 環境信息
- 瀏覽器: Chrome 118.0.5993.70
- 作業系統: macOS 12.6
- 螢幕截圖: [DatePickerStyleIssue.png]
- 控制台錯誤: 
  ```
  Failed to load resource: net::ERR_FILE_NOT_FOUND 
  styles/datepicker.css
  ```

### 可能原因
日期選擇器的CSS文件路徑可能不正確或文件缺失。
```

這樣的缺陷報告提供了足夠的信息，讓開發人員能夠快速定位和修復問題。

## 手動測試的局限與解決方案

手動測試雖好，但確實有其局限性。作為未來的軟件工程師，你需要了解這些局限並找到平衡點：

### 時間成本與解決方案

**問題**：全面的手動測試耗時巨大。SADo 的每個功能點都手動測試可能需要數天。

**解決方案**：
- 採用風險導向測試，專注於核心功能和高風險區域
- 建立優先級系統，如 P0（必測）、P1（重要）、P2（次要）
- 使用測試矩陣確保核心功能的關鍵路徑被覆蓋

### 一致性問題與解決方案

**問題**：不同人執行相同測試可能得出不同結果。小明可能認為 SADo 的反應速度「足夠快」，而小華認為「太慢」。

**解決方案**：
- 制定明確的通過/失敗標準，如「待辦事項添加應在 1 秒內完成」
- 使用標準化的測試環境
- 定期校準測試團隊的評判標準

## 何時轉向自動化測試？

作為一名開發或測試工程師，你需要判斷何時投資自動化測試。對於 SADo 專案，以下情況應考慮自動化：

### 重複測試場景

如果你發現自己第 10 次測試「添加待辦事項」功能，這就是自動化的好候選者。

**SADo 自動化建議**：
- 基本的 CRUD 操作：新增、編輯、刪除待辦事項
- 登入/登出流程
- 篩選和排序功能的基本場景

### 回歸測試需求

當 SADo 已經有了穩定的功能集，而你需要確保新功能不會破壞現有功能時，自動化回歸測試將節省大量時間。

## 結合二者的最佳實踐

在 SADo 專案中，理想的測試策略是手動測試和自動化測試的結合：

```
功能開發 → 手動探索性測試 → 修復問題 → 編寫自動化測試 → 定期手動審查
```

舉例來說：

1. 開發團隊完成「待辦事項標籤」功能
2. 測試團隊進行手動探索性測試，發現幾個邊界問題
3. 修復這些問題後，編寫 Playwright 自動化測試覆蓋主要場景
4. 在後續迭代中，自動化測試作為回歸測試運行，而手動測試專注於新功能

## 小結：平衡的藝術

手動測試和自動化測試就像陰陽兩面，相互補充而非對立。對於 SADo 這樣的專案，你需要：

- 善用手動測試的靈活性和創造性，特別是在開發初期
- 逐步構建自動化測試套件，覆蓋穩定且重複的功能
- 持續評估與調整測試策略，適應專案的發展

記住，測試的最終目標是確保 SADo 應用能夠為用戶提供優質的體驗，而不是盲目追求測試自動化的比例。有時，一次深入的手動探索性測試會發現十個自動化測試腳本都未能發現的問題。

作為未來的工程師，掌握何時依賴人類的創造力和直覺，何時利用機器的效率和一致性，將是你測試技能的精髓所在。
